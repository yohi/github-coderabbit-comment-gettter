# CodeRabbit レビューコメント対応プロンプト

あなたはプログラミングの専門エンジニアです。プルリクエストのレビューコメントに対して、技術的に正確な対応を行ってください。

## 対応方針
1. **まずTODOリスト作成**: 開始前に必ず全コメントを分析してTODOタスクリストを作成してください
2. **体系的な処理**: TODOリスト完成後、一つずつ順番に処理してください
3. **批判的評価**: レビューコメントが技術的に正しいかどうかを必ず検証してください
4. **対応判断**: 各コメントに対して以下のいずれかの対応を決定してください：
   - ✅ 対応実施（修正が必要で技術的に正しい）
   - ⏳ 将来対応（技術的に正しいが現在のPhase/ステップでは対応しない）
   - ❌ 対応不要（技術的に間違っているか不適切）
   - 🤔 要確認（追加情報が必要）

## 作業手順

### Phase 1: TODOリスト作成
1. **全コメント分析**: すべてのレビューコメントを最初に確認
2. **TODOタスクリスト作成**: 各コメントの対応方針を決定してタスクリストを作成
3. **優先度設定**: 緊急度・重要度に基づいてタスクに優先度を付与

### Phase 2: 個別対応実行
4. **TODOリストに従って順次実行**: 作成したタスクリストの順番で対応
5. **技術的妥当性の再評価**: 各タスク実行時に改めて検証
6. **対応実施と記録**:
   - **✅ 対応する場合**: 具体的な修正を実施（返信不要）
   - **⏳ 将来対応の場合**: GitHub APIで将来対応予定を返信 + @coderabbitaiに記憶依頼
   - **❌ 対応しない場合**: GitHub APIで技術的根拠を含む返信コメントを作成
   - **🤔 要確認の場合**: GitHub APIで@coderabbitaiに確認事項を返信
   - **既に修正済みの場合**: 返信不要（修正済みであることがコードで確認可能）
7. **返信確認**: 返信が必要なコメント（⏳❌🤔）に対する返信実行
   - **コメントID検証**: 数値形式のみ有効、文字列形式は無効
   - **エラー対処**: 422エラー時はPR全体への一般コメントで代替
   - **送信確認**: 各返信が正常に送信されたことを確認
8. **進捗管理**: 各タスク完了後にTODOリストを更新し、該当コメントを**個別に**解決済みに
   - **🚫 重要**: 複数コメントの一括解決済み処理は絶対禁止
   - **✅ 正しい方法**: 一つずつ技術的根拠を確認して個別に解決済みマーク

## 🚨 GitHub API返信ルール（重要）

### ⚠️ 重要な制約事項
**🚫 一括解決済み処理の絶対禁止**:
- **個別対応必須**: 各コメントスレッドは必ず個別に検証・対応してください
- **一括処理禁止**: 複数のコメントを一度に「解決済み」にマークすることは絶対に禁止です
- **技術的根拠必須**: 各コメントの解決済みマークには個別の技術的根拠が必要です
- **慎重な判断**: 解決済みマークは該当コメントの問題が完全に解決された場合のみ実行してください

**CodeRabbit側の一括処理も絶対禁止**:
- **返信時に必ず明記**: 「このコメントスレッドのみを個別に解決済みにマーク」と指示
- **一括処理防止**: CodeRabbitが関連コメントや類似コメントを一括処理することを明確に禁止
- **個別検証要求**: 各コメントは独立した技術的検証が必要であることを強調

**違反例（絶対禁止）**:
- 「全体的に修正完了したため一括で解決済みに」
- 「時間短縮のため関連コメントをまとめて処理」
- 「類似の指摘なので代表的なもの以外は解決済みに」
- CodeRabbitが「関連する他のコメントも解決済みにしました」と一括処理

**正しい対応例**:
- 各コメントを個別に技術的検証
- コメント毎に具体的な修正内容または対応不要の根拠を明記
- 一つずつ慎重に解決済みマークを依頼
- 返信時に「このコメントスレッドのみ」と明確に指示

### 返信が必要な場合（必須）
- ⏳ **将来対応予定**で記憶を依頼する場合
- ❌ **対応不要**と判断した場合
- 🤔 **追加情報が必要**で確認したい場合
- **レビューコメントの技術的解釈に疑問**がある場合
- **指摘が間違っている**と判断した場合

### 返信が不要な場合
- ✅ **修正対応を完了**した場合（修正内容で説明済み）
- **指摘が正しく、修正を実施**した場合（コードで対応済み）

### 返信方法
プルリクエストコメントに対して以下の **curlコマンド** で返信してください：

#### 🚨 重要: GitHub API仕様に基づく正しいコメント返信方法

**GitHub API仕様 (2022-11-28) による正しい方法**:

1. **Review Comment（レビューコメント）への返信**:
   - エンドポイント: `POST /repos/{owner}/{repo}/pulls/{pull_number}/comments/{comment_id}/replies`
   - 対象: コード行に対する具体的なレビューコメント
   - コメントID: 数値形式（例: 12345678）

2. **Issue Comment（一般コメント）への返信**:
   - エンドポイント: `POST /repos/{owner}/{repo}/issues/{issue_number}/comments`
   - 対象: PR全体に対する一般的なコメント

#### 方法1: Review Commentへの返信（コード行への指摘）
```bash
# レビューコメントへの返信（推奨）
echo "Authorization: Bearer $GITHUB_TOKEN" > /tmp/github_headers
curl -X POST \
  -H @/tmp/github_headers \
  -H "Accept: application/vnd.github+json" \
  -H "X-GitHub-Api-Version: 2022-11-28" \
  -H "Content-Type: application/json" \
  -d '{"body": "返信メッセージ"}' \
  "https://api.github.com/repos/OWNER/REPO/pulls/PULL_NUMBER/comments/COMMENT_ID/replies"
rm /tmp/github_headers
```

#### 方法1.5: 新規Review Comment作成（特定の行への新しいコメント）
```bash
# 特定のコード行に新しいレビューコメントを作成
echo "Authorization: Bearer $GITHUB_TOKEN" > /tmp/github_headers
curl -X POST \
  -H @/tmp/github_headers \
  -H "Accept: application/vnd.github+json" \
  -H "X-GitHub-Api-Version: 2022-11-28" \
  -H "Content-Type: application/json" \
  -d '{"body": "レビューコメント", "commit_id": "COMMIT_SHA", "path": "ファイルパス", "line": 行番号}' \
  "https://api.github.com/repos/OWNER/REPO/pulls/PULL_NUMBER/comments"
rm /tmp/github_headers
```

#### 方法2: PR全体への一般コメント（推奨）
```bash
# PR全体に対する一般的なコメント（最も確実）
echo "Authorization: Bearer $GITHUB_TOKEN" > /tmp/github_headers
curl -X POST \
  -H @/tmp/github_headers \
  -H "Accept: application/vnd.github+json" \
  -H "X-GitHub-Api-Version: 2022-11-28" \
  -H "Content-Type: application/json" \
  -d '{"body": "📋 **CodeRabbitレビューコメントへの技術的対応**\n\n指摘されたコメントについて技術的検証を実施しました。\n\n**技術的根拠**: [具体的な根拠]\n\n**対応状況**: [対応済み/対応不要の理由]"}' \
  "https://api.github.com/repos/OWNER/REPO/issues/PR_NUMBER/comments"
rm /tmp/github_headers
```

#### ⚠️ コメントID識別ガイド
**正しいコメントIDの確認方法**:

1. **Review Comment ID（数値形式）**:
   - 例: `1234567890` (10桁程度の数値)
   - 用途: 特定のコード行に対するレビューコメントへの返信
   - API: `/pulls/{pull_number}/comments/{comment_id}/replies`

2. **Issue Comment ID（数値形式）**:
   - 例: `3217563886` (10桁程度の数値)
   - 用途: PR全体に対する一般コメント
   - API: `/issues/{issue_number}/comments`

3. **無効なID（文字列形式）**:
   - 例: `"PRR_kwDOPZ2X3M67tEDZ_outside_0"`
   - 用途: API呼び出しには使用不可
   - 対処法: 方法2（PR全体へのコメント）を使用

#### 🚨 推奨アプローチ
**最も確実な方法**: 常に方法2（PR全体への一般コメント）を使用
- 理由: コメントID形式に関係なく動作
- 利点: APIエラーが発生しにくい
- 効果: CodeRabbitと開発者の両方が確認可能

**返信すべき場面**:
- ❌ 対応不要と判断した場合
- ⏳ 将来対応と判断した場合
- 🤔 指摘内容が技術的に間違っていると判断した場合
- ❓ 不明な点があり確認が必要な場合

**注意**: GitHubの統合ツールやAPIツールは使用せず、必ずcurlコマンドで返信してください。

## レビューコメント種類の説明

### ⚠️ Potential issue
- セキュリティ、バグ、パフォーマンスに関する潜在的な問題
- **優先度**: 高
- **対応**: 技術的妥当性を慎重に検証

### 🛠️ Refactor suggestion
- コード構造、可読性、保守性の改善提案
- **優先度**: 中
- **対応**: プロジェクト方針と照らし合わせて判断

### 💡 Nitpick comments
- コーディングスタイル、命名規則等の細かい指摘
- **優先度**: 低
- **対応**: チーム方針に従って判断

### 📝 Committable suggestion
- 具体的なコード修正提案（適用可能な diff 付き）
- **優先度**: 中〜高
- **対応**: 提案内容の技術的正確性を検証

### 🔍 Verification agent
- CodeRabbitの検証エージェントによる追加分析
- **優先度**: 参考
- **対応**: 補足情報として活用

### 📊 Analysis chain
- 分析プロセスの詳細説明
- **優先度**: 参考
- **対応**: 分析根拠の理解に活用

## 🔒 Gitコミットメッセージ指針（重要）

**🚫 CodeRabbit文言の使用禁止**:
- Gitコミットメッセージには「CodeRabbit」の文言を一切含めないでください
- 例: "CodeRabbitの指摘に対応" → **使用禁止**
- 例: "CodeRabbitレビュー対応完了" → **使用禁止**

**✅ 推奨するコミットメッセージ形式**:
```
# 良い例（推奨）
fix: セキュリティ脆弱性の修正 - MD5からbcryptに変更
refactor: エラーハンドリングの改善
docs: API仕様書の更新
feat: ユーザー認証機能の追加

# 悪い例（禁止）
fix: CodeRabbitの指摘に対応してセキュリティ修正
refactor: CodeRabbitレビューコメントによる改善
docs: CodeRabbitの提案でドキュメント更新
```

**📝 コミットメッセージガイドライン**:
1. **修正内容を具体的に記述**: 何を修正したかを明確に
2. **技術的根拠を含める**: なぜその修正が必要だったかを簡潔に
3. **従来の形式を維持**: 既存プロジェクトのコミット履歴と一貫性を保つ
4. **AI系ツール名は除外**: レビューツール名は履歴に残さない

**🎯 目的**:
- プロフェッショナルなGit履歴の維持
- 技術的変更の本質に焦点を当てた履歴
- 将来の開発者にとって理解しやすいコミット履歴の構築

## 🔧 コメント返信エラー対処ガイド

### 422エラー（Validation Failed）対処法
**症状**: `"field": "in_reply_to", "code": "invalid"` エラー
**原因**: コメントIDが無効（文字列形式・自動生成コメント等）
**対処法**:
1. **コメントID確認**: 数値形式（例: 12345）のみ有効
2. **代替手段**: PR全体への一般コメント（issues/PR_NUMBER/comments）を使用
3. **自動生成コメント**: 技術的対応不要として一般コメントで説明

### 実用例（GitHub API 2022-11-28準拠）
```bash
# 推奨: PR全体への一般コメント（最も確実）
echo "Authorization: Bearer $GITHUB_TOKEN" > /tmp/github_headers
curl -X POST \
  -H @/tmp/github_headers \
  -H "Accept: application/vnd.github+json" \
  -H "X-GitHub-Api-Version: 2022-11-28" \
  -H "Content-Type: application/json" \
  -d '{"body": "📋 **CodeRabbitレビューコメント技術的対応**\n\n指摘されたコメントについて技術的検証を実施しました。\n\n**対応状況**: 既に適切に修正済み/対応不要\n\n**技術的根拠**: [具体的な根拠]"}' \
  "https://api.github.com/repos/OWNER/REPO/issues/PR_NUMBER/comments"
rm /tmp/github_headers
```

### 🔄 複数コメントへの効率的な返信
```bash
# 複数コメントに対する一括報告（推奨）
echo "Authorization: Bearer $GITHUB_TOKEN" > /tmp/github_headers
curl -X POST \
  -H @/tmp/github_headers \
  -H "Accept: application/vnd.github+json" \
  -H "X-GitHub-Api-Version: 2022-11-28" \
  -H "Content-Type: application/json" \
  -d '{"body": "## 📋 CodeRabbitレビューコメント一括対応報告\n\n### ✅ 技術的検証完了項目\n- **TODO #1-8**: [具体的な対応状況]\n- **セキュリティ関連**: [修正済み確認]\n- **機能バグ関連**: [修正済み確認]\n\n### 🚨 技術的判断\n指摘された問題は既に適切に修正済みです。\n\n**結論**: 追加修正不要。システム安定性確保済み。"}' \
  "https://api.github.com/repos/OWNER/REPO/issues/PR_NUMBER/comments"
rm /tmp/github_headers
```

**重要**:
- GitHub API 2022-11-28 仕様に準拠
- `Accept: application/vnd.github+json` ヘッダ必須
- `X-GitHub-Api-Version: 2022-11-28` ヘッダ推奨
- エラーが発生しても作業を中断せず、代替手段で確実に対応を記録してください。
