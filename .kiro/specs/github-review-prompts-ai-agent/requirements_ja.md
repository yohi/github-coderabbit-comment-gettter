# 要件定義書

## 概要

この機能は、既存のGitHubレビュープロンプト抽出ツールを拡張し、CodeRabbitのPRレビューからAIエージェント最適化プロンプトを作成します。このツールは未解決のレビューコメントを抽出し、適切なペルソナとコンテキストでフォーマットし、AIエージェントが各レビューコメントを個別に慎重に評価し、不正確または不要な提案の可能性を考慮しながら応答するよう導く構造化プロンプトを生成します。

## 要件

### 要件1

**ユーザーストーリー:** 開発者として、GitHubのPRからCodeRabbitのレビューコメントを抽出してAIエージェント用プロンプトに変換したい。そうすることで、AIの支援を受けてレビューフィードバックに効率的に対応できる。

#### 受け入れ基準

1. GitHub PR URLが提供された時、システムは未解決のCodeRabbitレビューコメントをすべて抽出する
2. コメントを抽出する時、システムはGitHubのGraphQL APIを使用して解決状況を正確に判定する
3. コメントが解決済みとマークされている時、システムはそれを出力から除外する
4. コメントが見つかった時、システムは元のコメント内容と位置情報を保持する

### 要件2

**ユーザーストーリー:** 開発者として、AIエージェント用プロンプトに適切なペルソナとコンテキストを含めたい。そうすることで、AIがより正確で文脈に関連した応答を提供できる。

#### 受け入れ基準

1. プロンプトを生成する時、システムはAIエージェント用の明確なペルソナ定義を含める
2. プロンプトをフォーマットする時、システムはファイルパスと行番号のコンテキストを含める
3. プロンプトを作成する時、システムは提案の慎重な評価の必要性を強調する
4. プロンプトを出力する時、システムは不正確または不要なコメントの処理に関する指示を含める

### 要件3

**ユーザーストーリー:** 開発者として、AIエージェントがレビューコメントを一つずつ処理するようにしたい。そうすることで、各コメントが集中的な注意を受け、重要な詳細を見落とすリスクを減らせる。

#### 受け入れ基準

1. プロンプトを生成する時、システムは順次処理を促す構造で出力する
2. 指示をフォーマットする時、システムはコメントを個別に対処すべきことを明示的に述べる
3. プロンプトを作成する時、システムは異なるレビューコメント間の明確な分離を含める
4. 指示を出力する時、システムは複数コメントの一括処理を推奨しない

### 要件4

**ユーザーストーリー:** 開発者として、AIエージェントがレビューコメントを却下する理由の正当化を提供するようにしたい。そうすることで、推論を理解し、コード品質基準を維持できる。

#### 受け入れ基準

1. プロンプトを生成する時、システムは却下理由の文書化に関する指示を含める
2. 出力をフォーマットする時、システムは却下応答のテンプレートを提供する
3. 指示を作成する時、システムはコメントが不正確または不要と判断された理由の説明を要求する
4. プロンプトを出力する時、システムは適切な却下文書化の例を含める

### 要件5

**ユーザーストーリー:** 開発者として、仮想環境管理にuvを使用してPython 3.13を使いたい。そうすることで、一貫した最新の開発慣行を維持できる。

#### 受け入れ基準

1. プロジェクトを設定する時、システムはPython 3.13をターゲットバージョンとして使用する
2. 依存関係を管理する時、システムは仮想環境管理にuvを使用する
3. プロジェクトを設定する時、システムは適切なpyproject.toml設定を含める
4. 依存関係をインストールする時、システムはパッケージ管理にuvを使用する

### 要件6

**ユーザーストーリー:** 開発者として、コメント解決検出の精度を向上させたい。そうすることで、すでに対処済みの問題に時間を無駄にしない。

#### 受け入れ基準

1. コメント解決状況をチェックする時、システムはヒューリスティック手法よりもGraphQL APIの結果を優先する
2. GraphQL APIが利用できない時、システムはREST API手法にフォールバックする
3. 解決状況が不確実な時、システムは除外よりも包含を優先する
4. コメントを処理する時、システムは多くのコメントがある大きなPRのページネーションを処理する

### 要件7

**ユーザーストーリー:** 開発者として、包括的なエラーハンドリングとログ機能が欲しい。そうすることで、問題をトラブルシューティングし、ツールの動作を理解できる。

#### 受け入れ基準

1. API呼び出しが失敗した時、システムはコンテキスト付きの明確なエラーメッセージを提供する
2. コメントを処理する時、システムは進捗と統計をログに記録する
3. 不正な形式のデータに遭遇した時、システムは他のコメントの処理を継続する
4. デバッグが必要な時、システムは詳細な診断情報を提供する

### 要件8

**ユーザーストーリー:** 開発者として、柔軟な出力オプションが欲しい。そうすることで、ツールを異なるワークフローに統合できる。

#### 受け入れ基準

1. 出力を生成する時、システムはコンソールとファイル出力の両方をサポートする
2. ファイルに保存する時、システムは国際文字サポートのためにUTF-8エンコーディングを使用する
3. 出力をフォーマットする時、システムは明確なマークダウン構造を使用する
4. プロンプトが見つからない時、システムは空の出力ではなく有益なフィードバックを提供する
